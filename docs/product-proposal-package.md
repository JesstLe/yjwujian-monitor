# 永劫无间藏宝阁监控系统 - 产品方案建议书

**版本：** 1.0
**日期：** 2026年3月7日
**项目名称：** 永劫无间藏宝阁价格监控系统
**目标受众：** 网易藏宝阁产品团队、游戏运营团队

---

## 1. 项目定位

本项目是一个针对《永劫无间》游戏官方藏宝阁的**第三方价格监控与对比工具**，核心定位为"游戏物品的股票盯盘系统"。通过实时跟踪藏宝阁物品价格变化，为玩家提供智能化的价格监控、历史趋势分析、物品对比和价格提醒功能。

**核心价值主张：**
- **实时监控：** 类似股票盯盘的实时价格跟踪，玩家无需手动刷新页面
- **智能筛选：** 基于稀有度、星级、谪星变体等复杂属性的精准筛选
- **价格预警：** 自定义目标价，系统自动推送通知
- **可视化对比：** 2D/3D 视图下的多物品横向对比，辅助购买决策
- **数据沉淀：** 价格历史记录与趋势分析，把握市场动态

**适用场景：**
- 玩家长期追踪心仪物品的价格走势
- 藏家批量监控多个物品的成交情况
- 商家竞品价格对比与市场分析
- 非高峰期价格低谷自动提醒，优化购买时机

## 2. 用户痛点分析

### 2.1 当前藏宝阁体验的局限性

基于对现有网易藏宝阁平台的观察，我们识别出以下用户痛点：

#### 2.1.1 缺乏主动价格跟踪
**现状：** 玩家需要手动搜索物品并记住价格，无法被动接收价格变动通知。
**影响：** 容易错过价格低谷，需要反复刷新页面查看价格更新。

#### 2.1.2 筛选能力有限
**现状：** 官方筛选主要基于价格区间和基础分类，无法针对高级属性进行精确筛选。
**影响：** 玩家难以快速找到符合特定星级组合或谪星变体属性的物品。

#### 2.1.3 物品对比体验不佳
**现状：** 只能通过标签页或新窗口打开多个物品，无法在同一视图内横向对比。
**影响：** 购买决策过程繁琐，难以直观比较多个候选物品的差异。

#### 2.1.4 缺乏历史数据
**现状：** 官方平台不提供物品价格历史记录，玩家无法判断当前价格是否合理。
**影响：** 缺乏数据支撑，玩家难以判断购买时机是否合适。

#### 2.1.5 无多渠道通知能力
**现状：** 价格变动只能通过访问平台发现，无法通过其他渠道（如微信、飞书等）实时提醒。
**影响：** 非在线时段容易错过优惠机会。

### 2.2 用户需求总结

| 用户类型 | 核心需求 | 期望体验 |
|---------|---------|---------|
| 收藏型玩家 | 长期追踪特定物品价格 | 自动监控，目标价提醒，历史趋势分析 |
| 投机型玩家 | 寻找价格低谷买入 | 实时价格变动推送，多条件筛选 |
| 商家 | 竞品价格对比 | 批量监控，价格预警，成交数据跟踪 |
| 新手玩家 | 辅助购买决策 | 物品对比功能，价格历史参考 |

## 3. 解决方案概述

本监控系统通过以下方式解决上述痛点：

### 3.1 核心架构

```
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│   前端界面      │◄────►│   Express API   │◄────►│   网易藏宝阁    │
│  (React + Vite) │      │  (TypeScript)   │      │     API         │
└─────────────────┘      └────────┬────────┘      └─────────────────┘
                                  │
                         ┌────────▼────────┐
                         │  SQLite 数据库  │
                         │  (WAL 模式)      │
                         └─────────────────┘
                                  │
                         ┌────────▼────────┐
                         │  监控定时任务    │
                         │  (node-cron)    │
                         └─────────────────┘
```

### 3.2 技术栈（已实现）

**后端：**
- Express.js + TypeScript
- SQLite (better-sqlite3) with WAL 模式
- node-cron 定时任务
- JWT 认证 + Cookie 存储
- Zod 数据验证
- Axios HTTP 客户端（带速率限制）

**前端：**
- React 18 + TypeScript
- Vite 构建工具
- React Router 路由管理
- TailwindCSS 样式
- JSZip 打包下载

**外部集成：**
- Bark 推送
- 飞书（Feishu）官方 SDK
- 钉钉（DingTalk）机器人
- PushPlus 微信推送
- 自定义 Webhook

### 3.3 工作流程

1. **用户注册/登录** → 系统创建个人空间
2. **搜索物品** → 通过 CBG API 获取物品列表
3. **添加监控** → 物品加入个人监控列表，设置目标价
4. **定时检查** → 后台定时任务（默认 5 分钟）检查价格更新
5. **触发提醒** → 价格低于目标价时，通过配置渠道推送通知
6. **查看趋势** → 查询价格历史数据，为后续趋势展示与决策提供依据
7. **物品对比** → 2D/3D 视图下对比多个物品

## 5. 精彩场景展示

### 5.1 场景一：价格低谷自动提醒

**用户故事：** 玩家小王想购买一款"殷紫萍·红皮"英雄皮肤，目标价格是 800 元。

**操作流程：**
1. 搜索"殷紫萍"，筛选"红皮"稀有度，设置价格区间 500-1000 元
2. 点击"添加监控"，设定目标价 800 元
3. 配置飞书推送通知
4. 系统每 5 分钟自动检查价格
5. 当该皮肤价格降至 750 元时，飞书收到推送通知："殷紫萍·红皮 已降至 ¥7.50，低于目标价 ¥8.00"
6. 小王立即登录平台，点击链接跳转到藏宝阁购买

**核心价值：** 解放玩家注意力，不错过任何优惠时机。

### 5.2 场景二：多物品对比决策

**用户故事：** 商家小李想进购一批武器皮肤，需要在 3 款相似商品中做决策。

**操作流程：**
1. 在搜索结果中找到 3 款候选武器皮肤
2. 点击"添加到对比"
3. 进入"对比页面"，2D 视图下横向对比价格、星级、变体属性
4. 切换到"3D 对比"视图，通过旋转对比物品外观
5. 根据综合对比结果选择最优商品
6. 可选：点击"导出截图"，将对比结果保存为图片

**核心价值：** 直观对比，辅助决策，提升购物效率。

### 5.3 场景三：谪星变体精准筛选

**用户故事：** 收藏家老张想寻找"颜色全解锁"的谪星武器。

**操作流程：**
1. 选择"武器皮肤"分类，筛选"红色"稀有度
2. 在高级筛选中勾选"变体解锁状态：全解锁"
3. 进一步设置星级过滤：3 星及以上
4. 系统自动过滤，仅显示符合条件的物品
5. 添加到监控列表，长期跟踪价格变化

**核心价值：** 精准筛选，满足高级玩家个性化需求。

### 5.4 场景四：价格趋势分析

**用户故事：** 投机型玩家小陈想判断当前是否是买入好时机。

**操作流程：**
1. 找到心仪物品，点击"详情"
2. 查看价格历史图表（30 天数据）
3. 观察价格走势，发现当前处于近期低谷
4. 结合历史平均价判断是否合理
5. 设置目标价并添加监控

**核心价值：** 数据驱动决策，避免盲目购买。

## 7. 模块化打包与交付方案

本系统采用模块化设计，可按需集成到官方平台。以下建议的交付方案：

### 7.1 方案 A：完整集成（推荐）

**描述：** 将监控系统作为官方平台的独立模块，深度集成用户系统与数据流。

**交付内容：**
1. 前端页面（监控列表、对比页面、设置页面）
2. 后端 API 服务（物品搜索、监控管理、通知服务）
3. 数据库脚本（表结构与索引）
4. CBG API 集成层（带速率控制）
5. 定时任务服务（价格监控）
6. 通知服务（飞书、钉钉、Bark 等）

**优势：**
- 功能完整，无需二次开发
- 用户体验一致
- 运维维护成本低

**交付周期：** 4-6 周（含测试与部署）

### 7.2 方案 B：API 服务集成

**描述：** 官方平台自行开发前端，仅提供后端 API 服务。

**交付内容：**
1. RESTful API 文档
2. 后端服务代码
3. 数据库脚本
4. API 集成指南

**优势：**
- 前端自由度更高
- 可深度适配官方 UI 风格
- 可复用官方认证体系

**交付周期：** 2-3 周

### 7.3 方案 C：核心功能移植

**描述：** 仅移植核心监控逻辑到官方平台，保留现有 UI。

**交付内容：**
1. 监控引擎核心代码
2. 价格检查逻辑
3. 通知服务代码
4. 集成技术文档

**优势：**
- 集成成本最低
- 对现有系统改动最小

**交付周期：** 1-2 周
